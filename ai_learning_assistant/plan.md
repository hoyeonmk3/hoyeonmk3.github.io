# AI 학습 어시스턴트 — 구축 계획 초안

> 국제대학 8개 학과별 다국어 RAG 기반 AI 어시스턴트
> 2단계 전략: Gemini API 프로토타입 → 로컬 LLM 전환

---

## 1. 프로젝트 목표

### 최종 목표
국제대학 홈페이지에 게재되어 **외부 누구나** 한국어·영어·중국어로 학과 정보를 질의할 수 있는 AI 어시스턴트 서비스.

### 운영 요건
| 요건 | 내용 |
|---|---|
| 접근성 | 홈페이지에서 누구나 사용 (로그인 불필요) |
| 비용 | **운영 시 토큰 비용 0원** (로컬 LLM) |
| 언어 | 한국어, 영어, 중국어 |
| 학과 | 8개 전체 |
| 응답 품질 | 학과별 공식 정보 기반, 환각(hallucination) 최소화 |

---

## 2. 2단계 전략

### Phase 1: Gemini API 프로토타입 (2-3주)

**목적**: 빠르게 동작하는 데모를 만들어 보직자 승인 획득 + 데이터/프롬프트 검증

```
[학과 데이터 수집] → [RAG 파이프라인 구축] → [웹 UI] → [데모 시연]
     1주                   1주                0.5주      0.5주
```

#### 1-1. 데이터 수집 (Week 1)
- **수집 대상** (학과당):
  - 교육과정표 (PDF/웹)
  - 학과 소개 페이지 텍스트
  - 입학 안내 / FAQ
  - 졸업 후 진로 정보
  - 교수진 소개
- **형식**: 학과별 폴더로 정리, 텍스트 추출 → Markdown/Plain text
- **담당**: 각 학과 행정실 협조 요청 or 학교 홈페이지 크롤링

#### 1-2. RAG 파이프라인 (Week 1-2)
- **Embedding**: Google `text-embedding-004` (다국어 지원)
- **Vector Store**: ChromaDB (로컬, 경량)
- **LLM**: Gemini 1.5 Pro (Google Ultra Pro 크레딧 활용)
- **구조**:
  ```
  사용자 질문 (중국어)
      ↓
  Embedding → Vector Search (학과별 필터)
      ↓
  관련 문서 상위 5개 검색
      ↓
  System Prompt + Retrieved Context + 질문 → Gemini API
      ↓
  다국어 답변 생성
  ```

#### 1-3. System Prompt 설계
```
당신은 [대학명] 국제대학의 [학과명] AI 학습 어시스턴트입니다.

역할:
- 해당 학과의 교육과정, 입학, 진로에 대해 정확하게 안내합니다.
- 사용자의 질문 언어(한국어/영어/중국어)에 맞춰 답변합니다.
- 제공된 학과 자료에 없는 내용은 "해당 정보는 학과 사무실에 
  문의해주세요"라고 안내합니다.

규칙:
- 다른 대학이나 학과에 대한 비교 질문에는 답하지 않습니다.
- 학과 자료에 근거한 답변만 제공합니다.
- 입학 상담, 장학금 등 민감한 사항은 공식 안내 링크를 제공합니다.
```

#### 1-4. 웹 UI (Week 2-3)
- **기술**: 정적 웹페이지 (HTML/CSS/JS) + API 백엔드
- **구성**:
  - 학과 선택 드롭다운 (8개)
  - 언어 선택 (한/영/중)
  - 채팅 인터페이스
  - 학과 로고 / 학교 브랜딩
- **호스팅**: 임시로 Google Cloud Run or 학교 서버

#### 1-5. 데모 시연 (Week 3)
- 보직자 대상 라이브 데모
- 시나리오: "중국어로 글로벌경영학과 졸업 후 진로 질문"
- 결과물: 스크린샷 + 데모 영상 + 접속 URL

### Phase 1 산출물
| 산출물 | 형태 |
|---|---|
| 동작하는 웹 데모 | URL |
| 8개 학과 데이터셋 | 정제된 텍스트/PDF |
| System Prompt | 검증된 프롬프트 세트 |
| 데모 시연 자료 | 영상 + 스크린샷 |

### Phase 2: 로컬 LLM 전환 (3-4주)

**목적**: Gemini API 의존 제거 → 자체 서버에서 무료 운영

```
[로컬 LLM 선정] → [서버 구축] → [마이그레이션] → [성능 검증] → [홈페이지 게재]
     0.5주           1주           1주            0.5주          1주
```

#### 2-1. 로컬 LLM 선정
- **후보** (다국어 성능 기준):
  | 모델 | 파라미터 | 다국어 | VRAM 요구 | 비고 |
  |---|---|---|---|---|
  | Qwen 2.5 | 7B / 14B | ★★★★★ (중국어 최강) | 6GB / 12GB | 중국 유학생 대상 최적 |
  | Llama 3.1 | 8B | ★★★★☆ | 6GB | 영어 기반, 다국어 양호 |
  | Gemma 2 | 9B | ★★★★☆ | 8GB | Google 생태계 호환 |
  | Phi-3 | 14B | ★★★☆☆ | 10GB | 영어 우수, 중국어 약함 |

- **추천**: **Qwen 2.5:14B** — 중국어 성능 최우수 + 한/영도 양호
  - 중국 유학생이 주 대상이므로 중국어 품질이 핵심

#### 2-2. 서버 구축
- **Ollama** 기반 로컬 추론 서버
- **하드웨어 요구**:
  | 구성요소 | 최소 | 권장 |
  |---|---|---|
  | GPU | RTX 3060 12GB | RTX 4070 12GB |
  | RAM | 16GB | 32GB |
  | Storage | 50GB SSD | 100GB SSD |
  | OS | Ubuntu 22.04 | Ubuntu 22.04 |
- **대안**: 학교에 GPU 서버가 없을 경우
  - 옵션 A: GPU 서버 1대 구매 (100-200만원) — 남은 예산 활용
  - 옵션 B: 학교 전산센터 기존 서버 활용
  - 옵션 C: CPU 전용 운영 (Qwen 7B + llama.cpp) — 속도 저하 감수

#### 2-3. 마이그레이션
- **변경점**:
  ```
  Phase 1                          Phase 2
  ───────                          ───────
  Gemini API                  →    Ollama (Qwen 2.5)
  Google text-embedding-004   →    nomic-embed-text (로컬)
  Google Cloud Run            →    학교 서버 (Nginx + FastAPI)
  ```
- **코드 변경**: API 엔드포인트만 교체 (LLM 호출부 추상화 설계)
- **데이터**: ChromaDB 그대로 유지, 임베딩만 재생성

#### 2-4. 성능 검증
- Phase 1 데모에서 사용한 질문 세트로 A/B 비교
- Gemini vs 로컬 LLM 답변 품질 비교 (다국어별)
- 응답 시간 측정 (목표: < 5초)

#### 2-5. 홈페이지 게재
- 학교 홈페이지에 iframe or 별도 서브도메인 연결
- SSL 인증서 적용
- 접근 로그 수집 (사용 통계 → 성과 보고용)

### Phase 2 산출물
| 산출물 | 형태 |
|---|---|
| 로컬 서버 운영 중 | 학교 인프라 |
| 홈페이지 게재 완료 | URL (공개) |
| Gemini vs 로컬 품질 비교 | 보고서 |
| 운영 비용 = 0원/월 | (전기료 제외) |

### 핵심 설계 원칙
1. **LLM 추상화**: Phase 1 → 2 전환 시 API 엔드포인트만 교체
2. **학과별 격리**: ChromaDB collection으로 학과 데이터 분리 → 오염 방지
3. **다국어 자동 감지**: 사용자 입력 언어를 감지하여 동일 언어로 응답
4. **출처 명시**: 답변에 참조한 문서/페이지 출처 표시 → 신뢰도
